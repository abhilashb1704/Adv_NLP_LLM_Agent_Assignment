Multi-Step LLM Agent · Grok API

Research Report
Generator

A six-step pipeline that turns any research topic into a polished Markdown report. Each step's output becomes the next step's input — no step can be removed without breaking the chain.

1
Decompose
LLM
Breaks the raw topic string into 3–4 focused, searchable sub-questions.
← topic: str
→ sub_questions[]
System Prompt
You are a senior research editor. Your job is to decompose a broad research topic into 3–4 focused, searchable sub-questions. Each sub-question must be specific enough that a web search would return useful results. Respond ONLY with a JSON object — no preamble, no markdown fences: { "topic": "<the original topic>", "sub_questions": ["<q1>", "<q2>", "<q3>"] }
Why: The JSON-only constraint makes parsing reliable. The "specific enough" instruction prevents vague sub-questions that return broad Wikipedia pages instead of targeted facts.
sub_questions[]
2
Web Search
TOOL
DuckDuckGo query per sub-question. Retrieves current info the LLM cannot produce from training alone.
← sub_questions[]
→ search_results{}
Tool Call
from ddgs import DDGS ddgs = DDGS() for q in state["sub_questions"]: hits = list(ddgs.text(q, max_results=4)) results_by_question[q] = [ {"title": h["title"], "snippet": h["body"], "url": h["href"]} for h in hits ]
Why a tool, not an LLM: Asking an LLM to invent search results is asking it to hallucinate. DuckDuckGo retrieves real, timestamped snippets. Error handling: if a query returns zero results, an empty list is stored and the chain continues.
search_results{}
3
Extract Facts
LLM
Filters noisy snippets into a clean JSON list of verifiable facts with source URLs.
← search_results{}
→ key_facts[]
System Prompt (excerpt)
Extract the most important, verifiable facts. Remove duplicates and irrelevant material. Respond ONLY with JSON: { "key_facts": [ { "sub_question": "<which sub-question>", "fact": "<one clear statement>", "source_url": "<URL or empty>" } ] } Include 6–12 key facts total.
Why separate from Step 4: Raw HTML snippets are noisy and redundant. A dedicated extraction step distils signal before synthesis begins. The mandatory source_url field makes every claim traceable.
key_facts[]
4
Synthesise Draft
LLM
Writes a structured 400–600 word Markdown draft from the structured facts.
← key_facts[]
→ draft: str
System Prompt (excerpt)
Write a coherent, well-organised research report draft. - 400–600 words - Use Markdown headings (## for sections) - Cover each sub-question in a dedicated section - Cite source URLs inline as [source](url) - End with a brief "## Summary" paragraph Do NOT add anything beyond the Markdown report.
Why: The word-count range prevents both 180-word sketches and 900-word essays. Mapping each section to a sub-question makes Step 5's critique structurally precise.
draft + key_facts[]
5
Critique
LLM
Adversarial peer-review of the draft against the original key facts.
← draft + key_facts[]
→ critique{}
Critique JSON Schema
{ "overall_quality": "brief 1-sentence verdict", "strengths": ["..."], "weaknesses": [ { "issue": "description", "suggestion": "concrete fix" } ], "missing_topics": ["..."], "unsupported_claims": ["..."] }
Why both draft AND key_facts: The critic cross-references claims against evidence, identifying unsupported statements — not just weak prose. Each weakness requires a suggestion to give Step 6 actionable instructions.
draft + critique{}
6
Revise → Final
LLM
Addresses every critique point. Produces 500–700 word publication-ready report with References section.
← draft + critique{}
→ final_report.md
System Prompt (excerpt)
Address every weakness and missing topic in the critique. Remove or qualify any unsupported claims. 500–700 words. Professional, neutral tone. Add "## References" listing all cited URLs. Output ONLY the final Markdown report.
Why separate from Step 4: Without a critique, revision is just rewriting. With the structured JSON critique, revision is targeted repair — every weakness.suggestion becomes an explicit fix instruction.
